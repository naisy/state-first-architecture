## 💡 State-First Architecture（SFA）とは？

SFAは、システムのあらゆる振る舞い、非同期プロセス、およびコンポーネント間の同期を**「状態（State）」と「遷移（Transition）」の論理モデルのみ**で決定論的に制御する設計思想です。

従来の開発でバグの温床になりがちだった「タイムスタンプによるイベントの前後判定」や「連打防止用のフラグ管理（`isSubmitting = true`）」といった、状態遷移図の外側にある隠れた変数を完全に排除します。

### SFAの3大原則
1. **状態遷移の絶対性**: システムの挙動は「現在の状態」と「発生したイベント」の交差点のみで決まる。アプリケーション層に暗黙の制御フラグやアドホックなif文は存在しない。
2. **役割（Role）の明示**: すべてのステートは、その所有者（Frontend / Backend / Worker）をスキーマ上に明記する。
3. **自律的な前方遷移**: 各状態は「過去に誰が自分をそうしたか」を知らない。ただ「次にどの条件でどこへ進めるか」という前方への可能性のみを握る。

## 🎮 概念実証（PoC）：SFAテトリス

本リポジトリには、SFAの強力さを証明するための実証デモとして**「ピュアJavaScriptで書かれたテトリス」**を同梱しています。

ゲーム開発は、リアルタイムなキー入力、衝突判定、アニメーションの割り込みなど、非同期と状態の管理が最も難しい分野の一つです。このデモでは、**ゲーム進行のハンドリングに関わる条件分岐を、状態定義（JSON）の設計だけでエレガントに解決**しています。

### SFAテトリスが証明する3つのこと

1. **アニメーション中の入力無視（連打バグの構造的排除）**
   ライン消去アニメーション（`FE_CLEARING`）状態の遷移定義には、移動キーや回転キーのイベントが記述されていません。そのため、ユーザーがキーをどれだけ高速連打しようが、エンジンが勝手にイベントを破棄します。アプリケーションロジック側には `if (isAnimating) return;` のような防衛フラグによる条件分岐は1行もありません。
   
2. **宣言的な「壁キック・床キック」の優先順位制御**
   「壁際や床で回転した時にミノを自動で押し出す」という複雑なキック挙動（SRS）を、配列の優先順位として定義（フォールバック遷移）しています。「その場で回せないなら左キック、それもダメなら右キック...」という仕様のルールが、ネストされた複雑なif/elseのジャングルではなく、JSONの並び順として視覚的に表現されています。

3. **ビューとロジックの完全な疎結合**
   テトリス画面の文字幅（全角・半角）の不一致によるレンダリングバグを修正した際、ゲームのコアロジック、状態定義、および衝突判定ガードには一切触れることなく、描画関数（`render`）の差し替えだけで安全に修正を完了しました。

## 🚀 デモの動かし方

1. 本リポジトリをクローンまたはダウンロードします。
2. `demo/index.html` をお好きなブラウザで直接開きます（インストールやビルドステップ、サーバーの起動は不要です）。

### 操作方法
* `Enter`: ゲーム開始 / リスタート
* `←` / `→`: ミノの左右移動
* `↑`: 時計回り回転
* `↓`: 反時計回り回転
* `Space`: ソフトドロップ（落下速度アップ）
