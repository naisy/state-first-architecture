# ステートファーストアーキテクチャ（SFA）技術仕様書

## 1. 概要と設計哲学

ステートファーストアーキテクチャ（SFA）は、システムが持つすべての振る舞い、制御、および役職（コンポーネント）間の同期を、**「状態（State）」と「遷移（Transition）」の論理モデルのみによって定義・解決する**ソフトウェア設計手法である。

SFAは、並行処理や非同期ネットワーク、ユーザーの高速連打といった「時間の不確実性」に対し、時系列（タイムスタンプ）や暗黙のカウンタ（リクエスト連番、二重送信防止用の真偽値フラグ）といった、状態遷移図の外側にある「隠れた変数」に依存することを許さない。すべての例外的な挙動や排他制御は、状態と遷移の記述の中に陽（明示的）にモデリングされ、エンジンによって決定論的に処理されなければならない。

### SFAの3大原則
1. **状態遷移の絶対性（Strict State-Driven）**
   システムのあらゆる挙動は、`現在の状態` と `発生したイベント` の組み合わせのみによって決定される。アプリケーション層において、状態の管理や進行制御のための暗黙のフラグおよびアドホックなif文は存在してはならない。
2. **役割（Role）の明示**
   すべてのステートは、それを物理的に所有・管理する役職（`frontend` / `backend` / `shared` 等）をスキーマ上に明記する。
3. **前方参照の自律性（Forward-Only Flow）**
   各ステートは、自らがどのような経路でその状態に至ったかを感知しない。ただ「自らが今どこにいて、次にどのイベントと条件（Guard）によって、どの状態（Target）へ進めるか」という「前方の遷移ルール」のみを自律的に保持する。

## 2. ステート定義メタモデル（共通規格スキーマ）

すべての状態は、以下のメタモデルに準拠した形式（JSON/YAMLなど）で記述され、開発プロセスにおける**「信頼できる唯一の情報源（SSOT）」**として扱われる。

```json
{
  "$schema": "[http://json-schema.org/draft-07/schema#](http://json-schema.org/draft-07/schema#)",
  "title": "SFA_StateDefinition",
  "type": "object",
  "properties": {
    "state_id": { 
      "type": "string", 
      "description": "一意なステート識別子。プレフィックスに役割を明記することを推奨（例: FE_IDLE, BE_PROCESSING）" 
    },
    "role": { 
      "type": "string",
      "description": "このステートを所有・実行するライフサイクル上の役割（例: frontend, backend, worker）"
    },
    "description": { 
      "type": "string", 
      "description": "この状態が意味するビジネスロジックの概要" 
    },
    "on_enter": { 
      "type": "string", 
      "description": "この状態に遷移した「直後」に実行される副作用（関数名）"
    },
    "on_exit": { 
      "type": "string", 
      "description": "この状態から脱出する「直前」に実行される副作用（関数名）"
    },
    "next_transitions": {
      "type": "array",
      "description": "この状態から遷移可能なルール群。配列の順序は『評価の優先順位』を意味する",
      "items": {
        "type": "object",
        "properties": {
          "event": { 
            "type": "string", 
            "description": "遷移をトリガーするイベント名。AUTOを指定した場合は、条件が満たされ次第自動遷移する" 
          },
          "guard": { 
            "type": "string", 
            "description": "遷移を許可するか判定する純粋関数名。省略時は常にtrueとみなす" 
          },
          "action": { 
            "type": "string", 
            "description": "この遷移が発生した瞬間に実行される、コンテキストを書き換える副作用関数名（任意）" 
          },
          "target_state": { 
            "type": "string", 
            "description": "遷移先のステートID" 
          }
        },
        "required": ["event", "target_state"]
      }
    }
  },
  "required": ["state_id", "role", "next_transitions"]
}
```

## 3. エンジン実行仕様（決定論的ルーティングアルゴリズム）

定義された論理モデルを解釈して動かす「SFAエンジン」は、以下の厳格なアルゴリズムに従って状態遷移をドライブしなければならない。

### ① 優先順位付きフォールバック走査 (Fallback Evaluation)
同一のイベントに対して、複数の遷移定義（異なるガード条件）が存在する場合、エンジンは**定義配列（`next_transitions`）の記述順（上から順）にすべてを走査**する。

1. イベント `E` が発生する。
2. 現在の状態の `next_transitions` から、`event === E` である候補を配列の順序通りに抽出する。
3. 各候補の `guard` 関数を順に評価する。
4. **「最初に `true` を返した候補」** を遷移先として確定し、走査を打ち切る（フォールバック成立）。
5. すべての候補のガードが `false` を返した場合、エンジンはイベント `E` を何も処理せずに**静かに破棄（Discard）**する。

### ② 自動（トランジェント）遷移の再帰的評価 (AUTO Transition)
状態遷移が完了した直後、新しい状態の定義に `event: "AUTO"` が含まれる場合、エンジンは次の入力を待つことなく即座にガード条件を評価し、自動遷移を開始する。このAUTO評価は、AUTOイベントを持たない、あるいはAUTOイベントのガードがすべて `false` になる状態（安定状態）に達するまで**再帰的に実行**される。

## 4. 非同期・時間不確実性の解決仕様

非同期ネットワークにおけるパケットの遅延や、フロントエンドでの多重クリックといった「時間の歪み」によるバグは、すべて**状態遷移の論理構造のみ**で解決する。

### ① 境界状態（Boundary State）による物理同期
異なる役職（例: Frontend と Backend）が通信を伴って同期する場合、送信側のシステムは必ず**「境界状態（同期待ち状態）」**を経由する。

*   **自動ロック（連打防止）**: 境界状態（例: `FE_WAITING_BE`）は、「BEからの成功通知（`SUCCESS`）」や「失敗通知（`ERROR`）」のイベントのみを遷移ルールに持つ。この状態では、ユーザーが画面上のボタンをどれだけ連打して他のイベント（例: `SUBMIT`）を発生させても、**境界状態にその遷移が存在しないため、すべてエンジンレベルで無視（破棄）される。** `isSubmitting` のような制御フラグをアプリケーションコードに導入してはならない。

### ② 状態の不一致による遅延イベントの「自然消滅」
ネットワークの遅延により、古い状態のときに発行されたイベントが遅れて到達した場合でも、システムの状態はすでに次の状態に遷移している。新しい状態の遷移テーブルにその古いイベントの記述がなければ、エンジンはそれを破棄する。時系列の比較（タイムスタンプ）を行わずに、**「状態の不一致」という論理的な盾によって古いパケットは自動的に無害化**される。

## 5. 開発者とシステムの「責務の完全分離」

SFAの最大の特徴は、アプリケーションの主たるロジック（ビジネスルールや画面遷移）から、**「状態管理のための条件分岐（if/switchマニア）」や「複雑な状態管理用のフラグ」の記述を排除できる点**にある。

幾何学的な物理ルールや、描画（UI）、パース用エンジンといった局所的なインフラストラクチャ層での条件分岐（if文）は許容されるが、システムの進行を制御するルールはすべて構造化データ（JSON/YAML）側に集約される。

```
+-------------------------------------------------------------+
|               1. 状態遷移定義 (JSON / YAML)                  |
|                 - システムの骨組み、ルール、優先順位          |
+-------------------------------------------------------------+
                               |
       +-----------------------+-----------------------+
       v                                               v
+-------------------------------+             +-------------------------------+
|     2. ガード (Guards)        |             |     3. アクション (Actions)    |
| - 状態遷移してよいかの検証      |             | - 遷移に伴う副作用の実行      |
| - 状態（データ）を汚さない純粋関数 |             | - UI描画、API通信、DB保存など |
+-------------------------------+             +-------------------------------+
```

1. **State definitions (JSON/YAML)**: システムの進行や遷移の仕様（骨組み）を宣言的に記述する。
2. **Guards (純粋関数)**: 状態を遷移させてよいかの「判定ルール（真偽値を返す）」のみを書く。
3. **Actions (手続き型関数)**: 状態の突入・脱出時、または遷移の瞬間に発生する「副作用（UIへの出力、外部API呼び出しなど）」のみを書く。

## 6. このアーキテクチャがもたらす検証可能性

すべての振る舞いが構造化データ（JSON/YAML）として記述されているため、プログラムを実行する前に**「静的解析（Static Analysis）」**のみで、システムのあらゆるバグを自動検証・保証することが可能となる。

*   **デッドエンド（死に状態）の検出**: どこからも遷移してこない孤立状態や、一度入ると二度と抜け出せないデッドエンド（意図しないもの）を、グラフ理論的解析で検出する。
*   **イベント網羅性の検証**: 「特定の状態でこのイベントが起きた場合の遷移先が定義されていない」といった、設計・仕様の考慮漏れを静的バリデーターが自動でアラートする。
*   **仕様書・状態遷移図の自動生成**: JSON定義から、常に最新でコードと100%一致することが保証された状態遷移図（Mermaid等）をビルド時に自動生成する。
